# Adversarial Review Report

## CRITICAL

None confirmed.

## HIGH

- `src/channel.ts:94`, `src/channel.ts:106`, `src/channel.ts:127`, `src/server.ts:544` — `channel.poll` can skip messages when provider history is paginated.
  Slack and Lark are fetched with `limit/page_size=50`, but the code ignores provider pagination and advances `inbound_cursor` to the max timestamp in the returned page. If more than 50 messages arrived since the previous cursor, older unreturned messages are now behind the advanced cursor and will never be ingested.
  Fix: page through `has_more` / provider page tokens until exhausted, or only advance to a cursor that proves all earlier messages were observed.

- `src/server.ts:621`, `src/server.ts:630`, `src/server.ts:635`, `src/server.ts:637` — `mirror.push` can produce split-brain duplicates under concurrent pushers.
  Two processes can see the same pre-existing `mirror_map` row with `linear_id IS NULL`, both run `findByMarker`, both see nothing, both call `createIssue`, and the last `UPDATE mirror_map` wins. One Linear issue is orphaned.
  Fix: add a DB-backed claim/lease state for pending mirror rows, or perform a post-create reconcile before storing the id. Do not allow two workers to create for the same `hub_id`.

- `src/server.ts:635`, `src/server.ts:637`, `src/server.ts:622` — crash recovery can falsely mark changed content as pushed.
  If the first create succeeded but the process crashed before recording `linear_id`, then the hub ticket changed, a retry finds the old Linear issue by marker and immediately writes the current hash without updating Linear. Future pushes skip because the hash matches, leaving Linear stale.
  Fix: when reconciling an existing issue by marker, always `updateIssue` with the current hub-derived title/body/state before advancing `last_pushed_hash`.

- `src/server.ts:150`, `src/server.ts:159`, `src/server.ts:163` — `save_issue` has a read-then-write race outside a transaction.
  Concurrent updates can last-write-win silently. The append-only `relatedTo` merge is especially vulnerable: two writers can both read the same old JSON, append different links, and the later update loses the first link.
  Fix: wrap ticket update read/merge/write in `BEGIN IMMEDIATE`, or express the merge as an atomic SQL update.

## MEDIUM

- `src/linear.ts:28`, `src/server.ts:647` — Linear provider error messages are trusted as secret-free and persisted to `events`.
  The code truncates GraphQL error text but does not scrub it. If Linear or an intermediary echoes a submitted token, variable, header-derived identifier, or sensitive payload in `errors[0].message`, it is returned/logged via `mirror.error`.
  Fix: never persist raw provider error text. Store a fixed error class/status, or scrub known secret patterns and request values before returning/logging.

- `src/server.ts:440`, `src/server.ts:447`, `src/server.ts:452` — `channel.register` stores caller-supplied `configRef` / `secretRef` without validating that they are env-var names.
  The design says tokens live only in env and DB stores only names, but the tool will happily persist `xoxb-...` or an app secret if the caller passes the value instead of the name.
  Fix: validate refs against an env-name regex such as `^[A-Z_][A-Z0-9_]*$`, and reject strings matching common token prefixes.

- `src/server.ts:537`, `src/server.ts:540`, `src/server.ts:544` — `INSERT OR IGNORE` can hide non-dedup insert failures and still advance the cursor.
  `OR IGNORE` suppresses more than uniqueness conflicts in SQLite, including some constraint failures. A malformed inbound message can be skipped while the cursor advances.
  Fix: use `ON CONFLICT(channel_id,direction,provider_msg_id) DO NOTHING`, and let all other insert errors rollback without cursor advancement.

- `src/server.ts:289`, `src/server.ts:290` — `doc.publish` can leave multiple `document_versions.status='current'`.
  Publishing v2 then v1 marks v1 current but never resets v2 to draft. The document row has one `current_version`, but the version ledger can contain multiple current rows.
  Fix: in a transaction, set all versions for the doc to `draft`, then set the chosen version to `current`.

- `src/server.ts:31`, `src/server.ts:38`, `src/server.ts:90` and `test/identity.ts:29` — identity checking cannot detect wrong-but-valid actor attribution.
  `DEVLOOP_ACTOR=qa` passes if `qa` exists, even when the pane was intended to be `dev`. The tests prove unknown/unset detection, not wrong-valid misattribution.
  Fix: make `identity-check` accept an expected actor/project and fail if resolved values differ; launcher/CI should call it with the expected identity or verify `whoami` through the actual MCP spawn.

## LOW

- `src/server.ts:391`, `src/db.ts:150` — repeated `topic.synthesize` in the same round throws a raw unique-constraint error.
  The rollback path is fine, but the tool result is not a clean conflict like `post.add`.
  Fix: pre-check for an existing synthesis in that round or catch the constraint and return a structured `CONFLICT`.

- `src/server.ts:92`, `src/server.ts:39` — `whoami` and `identity-check` return the DB path.
  Not a token, but it is local environment disclosure. Fine for operator diagnostics, questionable for normal MCP tool exposure.
  Fix: return a basename or `dbConfigured: true` unless full path is explicitly requested.

## NIT

- `test/docs.ts:40` — the CAS test is sequential stale-base testing, not true cross-process concurrency.
- `test/channel.ts:136` — channel cursor tests cover duplicate replay, but not provider pagination, malformed insert rollback, or two concurrent pollers.
- `test/mirror.ts:92` — mirror tests are dry-run only for tool behavior, so they do not prove real crash recovery, marker reconciliation, or concurrent mirror pushes.
- `test/identity.ts:37` — the unset-env case is acknowledged as a “signal”, but the test does not prove a launcher will compare it against the intended actor.

## Test Result

`npm test` passed.

## Verdict

Fix-first. The implementation is reasonably scoped and many project filters/transactions are present, but P6 pagination and P7 mirror recovery/concurrency can violate core “system of record” claims.

Highest-value fixes:

1. Implement provider pagination and safe cursor advancement in `channel.poll`.
2. Make `mirror.push` concurrency-safe and update reconciled Linear issues before advancing hashes.
3. Put ticket updates, especially `relatedTo` merge, behind an atomic transaction.
